                  Writing Model
                  Two different content strategies are used depending on the selected category.
                

              

              

                

                  

                    SEO
                    
Intent analysis, meta structure, H1/H2/H3 outline, long-tail coverage, conclusion, and FAQ.

                  

                  

                    GEO
                    
Answer-first layout, proof blocks, references, trust cues, and entity alignment.

                  

                  

                  Cache
                    
Each task stores one `keyword` field. In mode 1 it is the keyword; in mode 2 it is the outline. Cache reuse keys off category + keyword + mode type + normalized brand info.

                  Bearer auth

                  Reusable article cache

                  Call Flow
                  The UI follows the same order your own client should use.
                

              

              

                

                  
1. Exchange token

                  
curl -X POST http://127.0.0.1:8028/api/token \
  -H "Content-Type: application/json" \
  -d '{"access_key":"YOUR_ACCESS_KEY"}'

                

                

                  
2. Create task

                  
curl -X POST http://127.0.0.1:8028/api/tasks \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "category": "seo",
    "keyword": "portable charger on plane",
    "info": "Brand: VoltGo. Product: 20000mAh portable charger.",
    "provider": "openai",
    "word_limit": 1200,
    "include_cover": 1,
    "content_image_count": 2
  }'

                

                

                  
3. Get task result

                  
curl http://127.0.0.1:8028/api/tasks/TASK_ID \
  -H "Authorization: Bearer YOUR_TOKEN"

                

              

            


            

              

                

                  Structure
                  The page stays a thin demo client over the live FastAPI service.
                

              

              

                

                  
Frontend

                  
`app/web/` renders this page and `app/web/static/demo/demo.js` calls the live token and task endpoints directly.

                

                

                  
API

                  
`app/api/` exposes the task creation and task result endpoints. The temporary token exchange route is used by the demo to mint a bearer token first.

                

                

                  
Services

                  
`app/services/` handles auth signing, prompts, writing, caching, image generation, and async task execution.
